P128 [Normal]: 1. Осуществлял сбор, обработку и анализ информации для проектирования баз данных
P129 [Normal]: В ходе практики была проанализирована предметная область проекта EXOFLORA - веб-энциклопедии экзотических растений. Основными информационными объектами системы являются пользователи, категории растений, карточки растений, изображения растений и записи проверки материалов. Такой набор сущностей отражает реальные сценарии приложения: пользователь изучает каталог, просматривает подробную карточку, а администратор управляет наполнением и проверяет записи.
P130 [Normal]: На этапе анализа были выделены данные, необходимые для хранения: имя пользователя, электронная почта, пароль и роль; название категории; русское и латинское название растения; описание, страна происхождения, уровень редкости; ссылки на изображения; статус проверки и комментарий администратора. Эти данные связаны с функциональностью сайта: каталогом, поиском, фильтрацией, детальной страницей растения и административной частью.
P131 [Normal]: Для проектирования были определены связи между сущностями. Одно растение относится к одной категории, но одна категория может включать несколько растений. Один пользователь может быть автором нескольких записей о растениях. У одного растения может быть несколько изображений. Проверка записи связана одновременно с растением и пользователем-администратором, который выполняет модерацию.
P132 [Normal]: При анализе ограничений были выделены первичные ключи id для всех таблиц, уникальность email пользователя, обязательность ключевых полей, внешние ключи между таблицами plants, users, categories, plant_images и reviews. Такие ограничения нужны для сохранения ссылочной целостности и предотвращения ситуаций, когда изображение или отзыв существуют без связанной записи растения.
P133 [Normal]: 2. Проектировал базу данных на основе анализа предметной области, разрабатывал объекты базы данных и реализовывал их в СУБД
P134 [Normal]: На основе анализа была подготовлена реляционная модель базы данных. В проектной модели используются пять таблиц: users, categories, plants, plant_images и reviews. Центральной таблицей является plants, так как именно она хранит основные сведения об экзотических растениях. Остальные таблицы расширяют модель: users отвечает за учетные записи и роли, categories группирует растения, plant_images хранит изображения, reviews фиксирует результаты проверки материалов.
P136 [Normal]: Рисунок 1 - ER-диаграмма базы данных EXOFLORA
P137 [Normal]: SQL-скрипт создания структуры базы данных был оформлен в файле exotic_plants_database.sql. В нем последовательно создаются таблицы users, categories, plants, plant_images и reviews, после чего добавляются демонстрационные записи. Для таблиц используются типы SERIAL, VARCHAR, TEXT и INT. Тип SERIAL выбран для автоматической генерации идентификаторов, VARCHAR - для строк ограниченной длины, TEXT - для длинных описаний и комментариев, INT - для внешних ключей.
P138 [Normal]: Таблица users содержит поля id, username, email, password и role. Поле email имеет ограничение UNIQUE, чтобы одна электронная почта не использовалась несколькими пользователями. Поле role позволяет разделять обычных пользователей и администратора. В демонстрационных данных добавлены роли admin и user.
P139 [Normal]: Таблица categories содержит справочник категорий растений. Таблица plants хранит основные карточки растений: name, latin_name, description, origin_country, rarity_level, category_id и created_by. Поля category_id и created_by являются внешними ключами и связывают растение с категорией и пользователем, который добавил запись.
P140 [Normal]: Таблица plant_images связывает изображения с конкретными растениями по plant_id. Это позволяет хранить несколько изображений для одной карточки. Таблица reviews содержит plant_id, admin_id, status и comment, поэтому может фиксировать состояние проверки записи: approved, pending или другое значение, принятое в учебной модели.
P141 [Normal]: В серверной части проекта также подготовлен файл exoflowers-server/sql/schema.sql, который описывает упрощенную таблицу plants для работы демонстрационного API. Backend реализован так, чтобы приложение могло запускаться как с PostgreSQL через DATABASE_URL, так и без базы данных через fallback-массив data/plants.js. Такой подход позволил сохранить работоспособность проекта во время разработки и постепенно переходить к полноценной базе данных.
P142 [Normal]: 3. Администрировал базу данных и описывал защиту информации в базе данных
P143 [Normal]: Для локального варианта проекта база данных предполагается к развертыванию в PostgreSQL. Подключение к базе настраивается через переменную окружения DATABASE_URL в формате postgres://user:password@localhost:5432/exoflowers. Серверное приложение читает эту переменную в db.js, создает пул подключений pg.Pool и выполняет SQL-запросы через общий метод query.
P144 [Normal]: Администрирование базы данных включает создание БД exoflowers, выполнение SQL-скрипта, проверку структуры таблиц, заполнение демонстрационными данными и проверку запросов SELECT, INSERT, UPDATE, DELETE и SELECT с JOIN. Эти операции были отражены в материалах проекта и позволяют проверить, что таблицы связаны корректно, а данные можно получать как отдельно, так и совместно через внешние ключи.
P145 [Normal]: Для разграничения доступа в учебной модели предусмотрены роли пользователей. Обычный пользователь может работать с каталогом и просматривать сведения о растениях, а администратор связан с проверкой и управлением записями. На уровне базы это отражается полем role в таблице users и связью reviews.admin_id с users.id. В промышленной версии для безопасности необходимо хранить не открытые пароли, а хеши паролей, а также выдавать приложению отдельного пользователя БД с минимально необходимыми правами.
P146 [Normal]: Защита целостности данных обеспечивается первичными и внешними ключами. Первичные ключи однозначно идентифицируют записи, а внешние ключи не позволяют связать растение с несуществующей категорией, изображение - с несуществующим растением, а отзыв - с несуществующим растением или администратором. Для предотвращения дублирования учетных записей используется уникальность email.
P147 [Normal]: Для безопасной передачи данных между приложением и базой используется серверный backend как промежуточный слой: клиент не подключается к PostgreSQL напрямую, а отправляет HTTP-запросы к Express API. Это скрывает строку подключения от браузера и позволяет централизованно обрабатывать ошибки, фильтры и операции CRUD. При публикации проекта переменные окружения должны храниться вне репозитория, а соединение с базой данных должно выполняться по защищенному каналу, если база размещена на удаленном се
P148 [Normal]: Резервирование данных для PostgreSQL может выполняться стандартными средствами pg_dump и pg_restore. Для учебного проекта также предусмотрен резервный механизм на уровне приложения: если DATABASE_URL не задан, backend использует локальный массив данных, что позволяет демонстрировать каталог без запущенной СУБД. Однако такой fallback не заменяет полноценное резервное копирование, потому что изменения в памяти процесса не сохраняются после перезапуска сервера.
P149 [Normal]: В результате практики была подготовлена структура базы данных для проекта EXOFLORA, определены таблицы, поля, типы данных, связи и ограничения, выполнено описание SQL-скрипта, рассмотрены вопросы доступа, ролей, целостности и резервного копирования. Полученная модель может быть использована как основа для дальнейшего расширения проекта: полноценной авторизации, хранения изображений, модерации материалов и переноса демонстрационных данных в PostgreSQL.
