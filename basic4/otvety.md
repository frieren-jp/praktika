# Ответы для устного опроса

## 1. Что такое абстрактный класс?

Абстрактный класс - это класс, который задает общий контракт для наследников. Он говорит: у всех реализаций должны быть такие-то методы.

В Python абстрактные классы обычно делают через `ABC` и `@abstractmethod`.

```python
from abc import ABC, abstractmethod


class UserRepoABC(ABC):
    @abstractmethod
    def get_by_id(self, user_id: int) -> str:
        pass
```

От обычного класса он отличается тем, что если не реализовать все абстрактные методы, объект создать нельзя. Будет `TypeError`.

## 2. Объясни Dependency Inversion своими словами

Dependency Inversion означает, что бизнес-логика не должна напрямую зависеть от конкретных технических классов.

Плохо:

```python
class UserService:
    def __init__(self) -> None:
        self.repo = PostgresUserRepo()
```

Здесь сервис сам создает `PostgresUserRepo`, поэтому он жестко привязан к PostgreSQL.

Хорошо:

```python
class UserService:
    def __init__(self, repo: UserRepoABC) -> None:
        self.repo = repo
```

Теперь сервису можно передать любую реализацию: настоящую для продакшена или мок для тестов.

## 3. Что такое `@abstractmethod`?

`@abstractmethod` помечает метод, который обязан быть реализован в дочернем классе.

Если класс не реализовал абстрактный метод, создать экземпляр не получится:

```python
repo = UserRepoABC()
# TypeError: Can't instantiate abstract class ...
```

## 4. Зачем разделять интерфейс и реализацию?

Интерфейс описывает, что нужно уметь делать. Реализация описывает, как именно это делается.

Разделение нужно, чтобы код было проще менять и тестировать. Например, `UserService` может зависеть от `UserRepoABC` и не знать, где реально лежат данные: в PostgreSQL, памяти или мок-объекте.

## 5. Пример нарушения Single Responsibility

Плохо:

```python
class UserManager:
    def save_user(self): ...
    def send_email(self): ...
    def validate_password(self): ...
```

У класса сразу несколько причин для изменения: база данных, отправка писем, правила валидации.

Лучше разделить:

```python
class UserRepo:
    def save(self, user): ...


class EmailService:
    def send_welcome(self, user): ...


class PasswordValidator:
    def validate(self, password: str): ...
```

Так каждый класс отвечает за одну область.

## 6. Что такое композиция?

Композиция - это когда один объект содержит другой объект и использует его. Это отношение `has-a`.

```python
class UserService:
    def __init__(self, repo: UserRepoABC) -> None:
        self.repo = repo
```

`UserService` имеет репозиторий, но не является репозиторием.

Композиция часто лучше наследования, потому что меньше связывает классы между собой. Зависимость можно заменить без переписывания всей иерархии.

## 7. Как аннотации типов связаны с Dependency Injection?

Аннотации типов показывают, какая зависимость нужна классу.

```python
class UserService:
    def __init__(self, repo: UserRepoABC) -> None:
        self.repo = repo
```

DI-контейнер может увидеть `repo: UserRepoABC`, найти подходящую зарегистрированную реализацию и передать ее автоматически.

Поэтому в архитектурных инструментах аннотации - это не просто подсказки для IDE, а способ понять, какие зависимости нужно собрать.

## Короткая схема ответа

Если забыл точное определение, отвечай так:

1. Какую проблему решает принцип или механизм.
2. Как это выглядит в коде.
3. Почему это полезно для тестов, замены реализации или поддержки проекта.

Пример:

Dependency Inversion решает проблему жесткой связи. Вместо того чтобы создавать `PostgresUserRepo` внутри `UserService`, мы передаем в сервис `UserRepoABC`. Тогда в тестах можно подставить мок, а в продакшене настоящую реализацию.
